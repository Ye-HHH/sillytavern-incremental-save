# 增量保存：源码修改 vs 纯插件 对比

## 源码修改版（当前部署的方案）

用户点发送 ──> AI生成回复完成 ──> saveChat() 被调用
                                        │
                                        ├─ 检查 hash：旧消息被改过吗？
                                        │    │
                                        │    ├─ 没改 → 【快速路径】
                                        │    │    │
                                        │    │    ├─ 只取新消息：chat.slice(549, 550)  ← 引用1条消息，几乎不占内存
                                        │    │    ├─ JSON.stringify([chatHeader, newMessage])  ← 序列化 6KB
                                        │    │    ├─ fetch(/save-append, 6KB)               ← 传输 6KB
                                        │    │    ├─ 更新 lastSavedHash
                                        │    │    └─ return                                 ← 全程 < 50ms
                                        │    │
                                        │    └─ 改了 → 回退全量（罕见）
                                        │
                                        ⏱ 耗时：几乎全部在 await fetch 网络往返
                                        📦 内存：几乎不产生临时大对象


## 纯插件版（假设基于拦截 fetch）

用户点发送 ──> AI生成回复完成 ──> saveChat() 被调用（插件改不了这个函数内部）
                                        │
                                        ├─ saveChat 照常运行（插件碰不到这里）
                                        │    │
                                        │    ├─ JSON.stringify([chatHeader, ...全部550条消息])
                                        │    │    │
                                        │    │    ├─ 遍历 550 条消息对象
                                        │    │    ├─ 每条消息包含 mes、swipes、extra 等字段
                                        │    │    ├─ 创建 186MB 临时字符串              ← 🔴 卡点1
                                        │    │    └─ 浏览器内存分配 186MB 连续空间
                                        │    │
                                        │    ├─ compressRequest() 包装请求
                                        │    │
                                        │    ├─ fetch(/api/chats/save, 186MB)  ← 发起全量请求
                                        │    │    │
                                        │    │    └─ [插件拦截 fetch]              ← 🔴 卡点2：到这里才发现是全量
                                        │    │         │
                                        │    │         ├─ 检查：是否可以走增量？
                                        │    │         ├─ 可以 → fetch(/save-append, 6KB)  ← 发增量
                                        │    │         ├─ 阻止原始 186MB fetch           ← 丢弃请求
                                        │    │         └─ 186MB 临时字符串变为垃圾
                                        │    │              │
                                        │    │              └─ 🔴 卡点3：V8 GC 回收 186MB 字符串
                                        │    │                  在回收期间可能暂停 JS 线程 50-200ms
                                        │    │
                                        │    └─ 返回（此时 saveChat 流程结束）
                                        │
                                        ⏱ 总耗时：saveChat内部耗时 + 插件拦截开销
                                        📦 内存峰值：186MB（JSON字符串）+ 原始chat对象（可能1GB+）

## 卡点详解

### 🔴 卡点1：JSON.stringify(186MB)

│
│ 时间轴 ──────────────────────────────────────────────>
│
│ 源码版：    ██ 6KB
│            ~2ms
│
│ 插件版：    ██████████████████████████████████████████████████ 186MB
│            ~200-500ms（取决于消息复杂度和设备性能）
│
│  影响：用户感觉"发送后页面卡了一下"。对于 550 条消息、
│  每条含 World Info 注入的大型 JSON，序列化本身就是重操作。

### 🔴 卡点2：拦截时机太晚

│
│ plugin intercepts HERE
│       │
│  saveChat() {
│    const body = JSON.stringify(chat);  ← 186MB 已经创建了
│    ...
│    fetch('/api/chats/save', body);     ← 插件才在这里拦截
│  }
│
│  影响：木已成舟，186MB 的内存分配已经发生。
│  插件只能阻止网络传输，但挡不住客户端的 CPU/内存消耗。

### 🔴 卡点3：GC 压力

│
│  每次全量 saveChat 调用：
│  ┌─────────────────────────────────────────────┐
│  │  1. 分配 186MB 字符串（JSON）                │
│  │  2. 分配请求对象、Headers、Body 等零散对象   │
│  │  3. 插件拦截，全量请求被丢弃                  │
│  │  4. 186MB 变为不可达（garbage）              │
│  │  5. V8 GC 运行，扫描 → 标记 → 清除           │
│  │     耗时 50-200ms，期间可能暂停所有 JS       │
│  └─────────────────────────────────────────────┘
│
│  你的场景：一次发送触发 3-5 次 saveChat
│  → 3-5 × (200ms 序列化 + 150ms GC) = 1-1.7秒 纯浪费
│
│  源码版：每次只序列化 4KB-30KB，GC 几乎无感知

### 🔴 卡点4：isChatSaving 排队效应

│
│  saveChat#1  ↑ 186MB序列化中... (200ms)
│               │ 插件拦截 → 发6KB增量 (50ms)
│               │ 186MB GC回收 (100ms)
│               ↓ isChatSaving = false
│
│  saveChat#2 ───────── 在排队等待 ───────────→ 终于拿到锁，重复上述过程
│  saveChat#3 ───────── 在排队等待 ──────────────────────────→ ...
│
│  源码版：saveChat#1 增量 50ms 完成，锁立即释放，#2/#3 马上接上

## 总结

│                     │  源码修改    │  纯插件      │  无优化      │
│  ──────────────────┼─────────────┼─────────────┼─────────────│
│  网络传输           │  6KB        │  6KB ✓      │  186MB       │
│  JSON序列化         │  6KB (2ms)  │  186MB (200ms)│ 186MB (200ms)│
│  GC压力             │  近乎零     │  高（反复触发）│ 高          │
│  isChatSaving锁     │  短         │  中等        │  长          │
│  多并发保存         │  流畅       │  微卡        │  严重卡顿    │
│  ──────────────────┼─────────────┼─────────────┼─────────────│
│  安装风险           │  改源码     │  零风险      │  零风险      │
│  升级兼容性          │  需适配     │  跨版本      │  跨版本      │
│  ──────────────────┼─────────────┼─────────────┼─────────────│
│  综合评价           │  ★★★★★     │  ★★★☆☆      │  ★☆☆☆☆      │

结论：
  源码修改 = 事前介入，在 stringify 之前就决定用增量
  纯插件   = 事后拦截，等全量序列化完成才发现"啊早就该用增量"
  
  插件版能解决 90% 的痛点（网络传输），但解决不了客户端的序列化+GC 内耗。
  186MB 的聊天文件，插件版每次保存还是会多花 200-400ms 在无意义的 CPU 工作上。
